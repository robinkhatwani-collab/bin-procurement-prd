Problem Definition

Why multi-party project tracking fails without a unified agentic platform

Background

Organisations managing complex, multi-party projects — procurement cycles, card launches, regulatory workflows — coordinate across dozens of sequential and parallel steps involving multiple stakeholder groups. Without a unified platform, these projects live in disconnected tools: static Excel or Smartsheet files, siloed Jira boards, or scattered Confluence pages — with no cross-tool visibility, no dependency intelligence, and no automated risk detection.

Reference Example — BIN Procurement: Launching a new payment card requires procuring a Bank Identification Number (BIN) from the card network. This is a complex, multi-party process involving the Issuer, Network, Processor, and Embosser — 31 sequential and parallel steps spanning ~5 months, currently managed in a static Excel file with no automation.

Solution Definition

A multi-agent web application — four specialised AI agents collaborating over a shared project data layer

Product Vision

An AI-Powered Multi-Project Tracker built as a web application. Users upload an Excel sheet (with DRI names, emails, steps, dates, and dependencies) and Agent 1 instantly generates a live HTML Status Tracker page. A central dashboard shows all active projects. Three additional agents — a chatbot (Agent 2), a deadline monitor (Agent 3), and a dependency evaluator (Agent 4) — run continuously over all tracked projects.

POC Scope: In the POC phase, HTML pages and authentication are managed by the user locally. No backend database is required. Excel is the data source. The future phase replaces Excel with Smartsheet via live API and adds a proper database and auth layer.

Agent Specifications

Four specialised AI agents — each with a distinct role, inputs, outputs, and agentic behaviour

1Source Configurator & Tracker Generator

Trigger: User uploads Excel file via web UI.

  • Validates schema (required columns A–H present)
  • Parses all steps, DRI names, emails, dates, statuses, dependencies
  • Computes RAG status, % complete, overdue count
  • Generates a self-contained HTML Status Tracker page
  • Adds project card to the central dashboard with a live link
  • Flags security steps (Column J = Y) — AI monitoring disabled for those rows

Agentic behaviour: Schema validation with intelligent error messaging (tells user which column is missing/malformed and why). Adaptive layout — if fewer than 10 steps, uses card view; 10+ steps uses table view.

Output: project-{slug}.html added to dashboard.

2Project Intelligence Chatbot

Trigger: User types a natural language question in the chat panel.

  • Has read access to all project HTML tracker pages visible on the dashboard
  • Answers queries across single or multiple projects
  • Supports multi-turn follow-up (remembers conversation context)
  • Can answer: "Which projects are at risk?", "Who owns the blocked step in Project X?", "What's overdue across all projects?"
  • Strictly grounded — never extrapolates beyond project data
  • Responds in ≤3 sentences unless a list is explicitly requested

Agentic behaviour: Plans a sequence of tool calls to answer complex queries (query Project A → query Project B → synthesise answer). Does not answer in one static prompt; it decomposes the question first.

Limitation (POC): Reads from generated HTML snapshots, not live Excel. Refresh required for latest data.

3Progress Monitor & Email Notifier

Trigger: Scheduled run (nightly) or on-demand by PM.

  • Reads all project Excel files; compares estimated end dates to today
  • Identifies: overdue steps, steps due within 3 days (warning), steps due within 7 days (heads-up)
  • Generates a personalised email for each affected DRI (name + email from Column C/D)
  • Sends a weekly project health summary to the PM / exec sponsor
  • Logs every email sent with timestamp, recipient, and step reference
  • Respects security flag — never emails about flagged steps without PM approval

Agentic behaviour: Observes project state → reasons about urgency and audience → decides whether to send immediate alert, warning, or include in weekly digest → acts (sends email). This is the core agentic loop: observe → reason → act.

Output: Emails sent; email log updated on dashboard.

4Dependency Evaluator & Milestone Planner

Trigger: On project load and after each status update.

  • Reads dependency graph from Column I (Depends On)
  • Identifies all steps currently blocked by incomplete upstream steps
  • Detects cascade risk — if Step N is delayed, which downstream steps are at risk?
  • Recommends the next 3–5 milestone steps the team should focus on
  • Highlights critical path (longest dependency chain to completion)
  • Generates a plain-language dependency risk summary for the PM

Agentic behaviour: Multi-step planning — reads the dependency graph, traverses the chain, computes critical path, reasons about current state, then outputs a prioritised action plan. Memory: retains prior milestone suggestions to detect regressions.

Output: Dependency risk panel on dashboard + recommended milestones widget.

Prompt Strategy

Structured, auditable prompts for Agents 2, 3, and 4 — grounded, scoped, role-aware

Design Principles

Grounded — always use real project data, never hallucinate. Scoped — output length and format explicitly constrained. Role-aware — audience and tone vary by recipient. Auditable — every AI output logged with its input context.

Agent 2 — Project Chatbot

SYSTEM: You are a helpful project assistant with read access to all projects tracked on the Multi-Project Tracker platform. You will be given live project data as JSON context. Answer only based on the provided data. If information is not in the data, say "I don't have that information." Never guess or extrapolate. Keep answers under 3 sentences unless the user asks for a list. You may query multiple projects to answer a question — plan your tool calls before responding. USER: [Project data snapshots: {{project_snapshots_json}}] User question: {{user_question}}

Agent 3 — DRI Deadline Alert Email

SYSTEM: You are a professional project coordinator. Write a brief, friendly, and professional reminder email to a stakeholder about an overdue or upcoming project step. Keep it under 100 words. Do not use blame language. Be specific about the step, deadline, and the downstream impact of the delay. Sign off with the PM's name. USER: Recipient: {{dri_name}} ({{dri_email}}) Project: {{project_name}} Step: {{step_number}} — {{step_description}} Was Due: {{estimated_end_date}} Days Overdue: {{days_overdue}} Next Dependent Step: {{next_step}} (owned by {{next_dri_name}}) PM Name: {{pm_name}} Write the email body only. No subject line.

Agent 3 — Weekly Executive Summary

SYSTEM: You are a senior program manager writing a weekly project status update for an executive audience. Write in clear, confident business prose. No jargon. Highlight RAG status. Maximum 200 words. USER: Week Ending: {{week_end_date}} Project: {{project_name}} Overall Status: {{rag_status}} Steps Completed This Week: {{completed_this_week}} Steps Overdue: {{overdue_steps}} Upcoming Next Week: {{upcoming_steps}} Dependency Risk Summary: {{agent4_risk_summary}} Key Blockers: {{blockers}} Write the executive summary. Start with overall health, then highlights, then risks, then ask.

Agent 4 — Dependency Risk & Milestone Planner

SYSTEM: You are a senior project risk analyst specialising in dependency management. Analyse the provided project dependency graph and current step statuses. Identify all blocked steps, cascade risks, and the critical path. Then recommend the next 3–5 milestones the team should prioritise. Be concise, data-driven, and actionable. Output as JSON. USER: Project: {{project_name}} Target Launch Date: {{launch_date}} Today's Date: {{today}} Step Statuses: {{step_statuses_json}} Dependency Graph: {{dependency_graph_json}} Provide: (1) blocked_steps (array), (2) cascade_risks (array with impact description), (3) critical_path (array of step numbers), (4) recommended_milestones (array of next 3–5 steps with rationale), (5) overall_risk_score 0–100.
