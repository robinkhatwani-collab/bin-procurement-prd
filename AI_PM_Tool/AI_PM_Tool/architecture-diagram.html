Deep-Dive: AI Agent & RAG Pipeline

Step-by-step internals of PRD generation and vector search

🤖
AI Agent — Project Page Generation
Triggered by POST /upload-project with a .xlsx file
1
Receive Excel Upload
Browser sends multipart POST with .xlsx, genPrd & genTracker flags. Server reads file bytes into a temp file.
POST /upload-project · multipart/form-data
2
Parse Excel · openpyxl
Reads metadata rows (project name, objective) then task rows (step description, DRI, due date, status, priority). Calculates progress % and unique DRI count.
parse_excel(tmp_path) → { title, tasks[], dris[], progress }
3
Slugify & Structure
Normalises project name into a URL-safe slug. Builds the data dict used by both generators. No external AI call — pure Python template logic.
slugify("BIN Procurement…") → "bin-procurement-for-new-card-launch"
4
Generate PRD · generate_prd()
Renders a full dark-theme HTML page with: hero stats, objective callout, stakeholder table, task milestone timeline, risk register, and acceptance criteria.
generate_prd(data, url_prefix) → prd.html string
5
Generate Status Tracker · generate_tracker()
Builds an interactive task board with status badges (Done / In Progress / Blocked / Not Started), DRI chips, progress bar, priority tags, and due dates.
generate_tracker(data) → status-tracker.html string
6
Save & Update Registry
Writes HTML files to projects/{slug}/. Updates registry.json with title, progress stats, and DRI count. Calls rebuild_projects_hub() to regenerate projects.html.
projects/{slug}/prd.html + status-tracker.html
7
JSON Response → Browser Redirects
Returns { success, prdUrl, trackerUrl, stepsCount, dris }. Browser shows confirmation and direct links to view the new PRD and tracker.
{ "success": true, "prdUrl": "/projects/..." }
🔍
RAG Pipeline — Chatbot Q&A
Phase A: indexing at server startup · Phase B: retrieval per question
Phase A — Indexing (once at server start · index_all())
1
Discover Project Pages
_discover_pages() scans projects/*/prd.html, projects/*/status-tracker.html, and Week1–6_Learning_Summary.html. Navigation pages (default, ai-learning) excluded — no useful content.
_discover_pages() → ["projects/bin-procurement.../prd.html", ...]
2
Parse & Strip HTML
BeautifulSoup removes all <script>, <style>, <nav>, <footer>, <head> tags. Only visible text content remains, preserving semantic structure.
BeautifulSoup(html) → clean text with headings intact
3
Section Chunking by Heading
Each h2/h3 heading = chunk boundary. Collects all text between current and next heading (≤1800 chars). Metadata per chunk: source, page_type, project name, section title.
{ text, source, page_type:"PRD", project:"BIN Procurement", section:"Problem Statement" }
4
Embed & Store in ChromaDB
ChromaDB embeds each chunk with ONNX MiniLM-L6-v2 (local, no API, 384-dim vectors). Persisted to chromadb_store/ with cosine-similarity HNSW index. Skips re-index if already loaded.
collection.upsert(ids, documents, metadatas) → ~30–60 chunks
Phase B — Retrieval (per user question · retrieve())
5
User Sends Question
Chat widget POSTs { message, history } to /api/chat. Server extracts the query string and conversation history (last 6 turns).
POST /api/chat → { message: "What's the BIN project status?" }
6
Semantic Search · Cosine Similarity
Query embedded with same ONNX model. ChromaDB returns top-5 chunks by cosine similarity. Each chunk includes its score (0–1) and full metadata.
collection.query(query_texts=[q], n_results=5) → chunks + scores
7
Build Scoped Prompt
Chunks formatted as "[PRD — Project | Section]\ntext". System prompt instructs Claude to answer ONLY from context and say "I don't have that information" if not found.
system: "Answer ONLY from context below. Never fabricate data."
8
Claude Haiku Generates Answer
Anthropic SDK calls claude-haiku-4-5-20251001 with system prompt + context + question. Max 1024 tokens. Conversation history included for multi-turn context.
client.messages.create(model="claude-haiku-4-5-20251001")
9
Return Answer + Source Citations
JSON { reply, sources[] } sent to browser. Chat widget renders response text and source chips showing page_type · project name with relevance score on hover.
{ reply: "...", sources: [{ page_type:"PRD", project:"BIN...", score:0.87 }] }
🔒

RAG Scope Enforcement — Two Layers

How the chatbot is prevented from answering outside project documents

🗄️ Layer 1 — Retrieval Scope (ChromaDB) Vector DB
  • Only project HTML pages are indexed — PRDs, status trackers, learning summaries
  • Navigation pages (default.html, ai-learning.html, projects.html) are explicitly excluded
  • _discover_pages() defines the exact whitelist — new pages must be added intentionally
  • Claude never sees content from outside this index — retrieval is the first gate
  • Re-index button in chat header refreshes after new projects are uploaded
✨ Layer 2 — Generation Scope (System Prompt) LLM Guard
  • "Answer ONLY from the project documents provided as context below"
  • "If the answer is not in the context, say: I don't have that information"
  • "Never fabricate data — always cite which project or document you reference"
  • "For status questions, reference actual numbers (tasks, %, DRIs) from context"
  • Source chips shown in UI make provenance visible to the user on every answer

Technology Stack

Every tool, its role, and why it was chosen

🐍
Python stdlib HTTP Server
server.py — zero-dependency local server. Handles GET (static), POST /upload-project, POST /api/chat, POST /api/reindex. Auto-installs deps at startup.
📊
openpyxl
Reads .xlsx project files. Extracts metadata rows (project name, objective) and task rows (step, DRI, due date, status, priority) for page generation.
🌿
BeautifulSoup4
RAG indexer HTML parser. Strips CSS/JS/nav/footer tags, extracts clean body text, and uses h2/h3 headings as semantic chunk boundaries.
🗄️
ChromaDB
Local persistent vector database. No external service. Stores HTML chunks with ONNX MiniLM embeddings. Cosine HNSW index. Persisted to chromadb_store/.
🔢
ONNX MiniLM-L6-v2
Sentence embedding model bundled with ChromaDB. Runs fully locally (22MB, no API call). Converts text chunks and queries into 384-dimensional vectors.
✨
Claude Haiku (Anthropic)
LLM for chat response generation via Anthropic SDK. Receives retrieved chunks as context. System prompt enforces strict scope. Requires ANTHROPIC_API_KEY.
🌐
Vanilla JS + CSS
Zero-framework frontend. Chat widget (FAB, panel, bubbles, source chips, loading dots), live clock, project hub — all plain JS. No build tools or bundlers needed.
📁
File System as Database
Generated HTML pages stored in projects/{slug}/. registry.json tracks all projects. No SQL DB needed — the file system is the source of truth for the project hub.
