Learning in Practice

How every major product decision in this PRD maps to a specific concept from Weeks 1–3

Week 1 · Blue
Foundations — Tokens, Transformers & RAG
▸Tokens & Context Windows — Agent 2's prompt budget is designed around Claude's 200K window. Project snapshots are compressed to JSON to stay under token limits.
▸RAG Architecture — Agent 2 uses RAG: project HTML pages are chunked, embedded, and retrieved before answering natural language queries. Never answers from training data.
▸Grounding — All four agent system prompts enforce strict grounding: answer only from provided project data. "I don't have that information" is a required fallback.
▸Encoder/Decoder Understanding — Chose Claude (decoder-only) for generation tasks. Encoder-style embeddings used for semantic search in Agent 2's retrieval layer.
Week 2 · Purple
Context Management & Agentic RAG
▸Agentic RAG — Agent 2 decomposes multi-project queries, plans multi-step retrieval tool calls, evaluates result quality, and loops — rather than a single-shot retrieval. Matches the Agentic RAG loop pattern.
▸Context Management (Good Design) — Agent 2 uses structured context objects (project JSON snapshots) rather than raw HTML dumps. Avoids "lost-in-the-middle" by keeping context lean and structured.
▸Context Types — Short-term: conversation history. Structured: project JSON. Unstructured: HTML tracker pages. Each type handled differently in the retrieval pipeline.
▸In-Context Learning — Agent 3 email prompts use few-shot examples of good vs. bad reminder tone to guide the LLM without fine-tuning.
Week 3 · Teal
Agentic AI, Evaluation & Responsible AI
▸Knowledge Graph → Agent 4 — Steps = nodes, dependencies = edges. Agent 4 traverses this graph to find critical path, blocked nodes, and cascade risks.
▸Perceive→Plan→Act Loop → Agent 3 — Observe (read deadlines) → Reason (classify urgency) → Act (send email). This is the agentic loop applied to project monitoring.
▸HHH Framework → Agent 3 guardrails — Helpful: timely alerts. Honest: only real data. Harmless: no blame language, never escalates without PM review.
▸Precision & Recall → Agent 4 metrics — P≥0.80 / R≥0.75 targets for cascade risk detection. F1 balance chosen deliberately — false negatives (missed risks) cost more than false positives.
▸Prompt Caching → Agent 2 cost — Repeated project context (scheme rules, dependency graph) is cached. Estimated 70–90% cost reduction on repeated multi-project queries.

Problem Definition

Why multi-party project tracking fails without a unified agentic platform

Background

Organisations managing complex, multi-party projects — procurement cycles, card launches, regulatory workflows — coordinate across dozens of sequential and parallel steps involving multiple stakeholder groups. Without a unified platform, these projects live in disconnected tools: static Excel or Smartsheet files, siloed Jira boards, or scattered Confluence pages — with no cross-tool visibility, no dependency intelligence, and no automated risk detection.

Reference Example — BIN Procurement: Launching a new payment card requires procuring a Bank Identification Number (BIN) from the card network. This is a complex, multi-party process involving the Issuer, Network, Processor, and Embosser — 31 sequential and parallel steps spanning ~5 months, currently managed in a static Excel file with no automation.

Solution Definition

A multi-agent web application — four specialised AI agents collaborating over a shared project data layer

W1 · RAG Architecture W2 · Agentic RAG W3 · Knowledge Graph → Agent 4 W3 · Prompt Caching

Product Vision

An AI-Powered Multi-Project Tracker built as a web application. Users upload an Excel sheet and Agent 1 instantly generates a live HTML Status Tracker page. Three additional agents — a chatbot (Agent 2), a deadline monitor (Agent 3), and a dependency evaluator (Agent 4) — run continuously over all tracked projects.

POC Scope
In the POC phase, HTML pages and authentication are managed by the user locally. No backend database is required. Excel is the data source. The future phase replaces Excel with Smartsheet via live API and adds a proper database and auth layer.

High-Level Architecture

W3 Learning Applied — Knowledge Graph as Dependency Layer
Agent 4's dependency model is a direct application of the Knowledge Graph pattern: steps are nodes, dependency relationships are typed edges, and critical path detection is graph traversal. This is exactly how a Knowledge Graph enables multi-hop reasoning — applied to project risk.

Agent Specifications

Four specialised AI agents — each designed around specific learning concepts from Weeks 1–3

1Source Configurator & Tracker Generator
W1 · Tokens W1 · Grounding

Trigger: User uploads Excel file via web UI.

  • Validates schema (required columns A–H present)
  • Parses all steps, DRI names, emails, dates, statuses, dependencies
  • Computes RAG status, % complete, overdue count
  • Generates a self-contained HTML Status Tracker page
  • Adds project card to the central dashboard with a live link
  • Flags security steps (Column J = Y) — AI monitoring disabled

Agentic behaviour: Schema validation with intelligent error messaging. Adaptive layout — card view under 10 steps, table view for 10+.

Output: project-{slug}.html added to dashboard.

2Project Intelligence Chatbot
W1 · RAG W1 · Grounding W2 · Agentic RAG W2 · Context Management W3 · Prompt Caching

Trigger: User types a natural language question in the chat panel.

  • Has read access to all project HTML tracker pages (RAG retrieval layer)
  • Decomposes complex queries into multi-step retrieval plans — Agentic RAG pattern
  • Maintains structured context state across turns — not raw conversation history
  • Supports: "Which projects are at risk?", "Who owns the blocked step in Project X?"
  • Strictly grounded — never extrapolates beyond project data (W1 Grounding)
  • Prompt caching applied to repeated project context — reduces query cost by ~70%

Agentic behaviour: Plans a sequence of tool calls to answer complex queries. Does not answer in one static prompt — decomposes the question first. (W2 Agentic RAG loop)

3Progress Monitor & Email Notifier
W2 · In-Context Learning W3 · Perceive→Plan→Act W3 · HHH Framework

Trigger: Scheduled run (nightly) or on-demand by PM.

  • Observe: reads all project Excel files, compares estimated end dates to today (Perceive)
  • Reason: classifies urgency — overdue / 3-day warning / 7-day heads-up (Plan)
  • Act: generates personalised DRI email, sends, logs (Act)
  • HHH guardrails: Helpful (timely), Honest (data-only), Harmless (no blame language)
  • Security flag respected — never emails about flagged steps without PM approval

Agentic behaviour: Observe → Reason → Act loop. This is the core agentic pattern from W3 Agentic Skills applied directly to project monitoring.

4Dependency Evaluator & Milestone Planner
W3 · Knowledge Graph W3 · SMART Goals W3 · Precision & Recall W3 · Agent Metrics

Trigger: On project load and after each status update.

  • Knowledge Graph traversal: Column I dependency data → nodes + edges → critical path (W3 Knowledge Graph)
  • Identifies all steps currently blocked by incomplete upstream steps
  • Detects cascade risk — if Step N delayed, which downstream steps are at risk?
  • Milestone recommendations follow SMART criteria: specific, measurable, time-bound (W3 SMART Goals)
  • Cascade risk target: P≥0.80, R≥0.75 — recall prioritised over precision (W3 Precision/Recall)

Agentic behaviour: Multi-step planning — reads graph, traverses chain, computes critical path, outputs prioritised action plan. Memory: retains prior milestone suggestions to detect regressions.

Precision vs Recall Trade-off Decision (Agent 4)
For cascade risk detection, we deliberately bias toward Recall (R≥0.75) over Precision. A false negative (missed cascade risk) causes a real project delay. A false positive (unnecessary alert) costs a PM 5 minutes of review. This is a direct application of the W3 Precision/Recall case study — the business cost asymmetry determines the dial setting.

Prompt Strategy

Structured, auditable prompts — built on W1 Grounding, W2 Context Management, W3 HHH Framework

W1 · Grounding W2 · Context Management W2 · In-Context Learning W3 · HHH Framework

Design Principles

Grounded (W1) — always use real project data, never hallucinate. Scoped (W2) — structured context objects, not raw document dumps. HHH-Aligned (W3) — every prompt enforces Helpful, Honest, Harmless. Auditable — every AI output logged with its input context.

Agent 2 — Project Chatbot W1 · RAGW2 · Agentic RAG

SYSTEM: You are a helpful project assistant with read access to all projects tracked on the Multi-Project Tracker platform. You will be given live project data as JSON context. Answer only based on the provided data. If information is not in the data, say "I don't have that information." Never guess or extrapolate. Keep answers under 3 sentences unless the user asks for a list. You may query multiple projects to answer a question — plan your tool calls before responding. USER: [Project data snapshots: {{project_snapshots_json}}] User question: {{user_question}}

Agent 3 — DRI Deadline Alert Email W3 · HHH — Helpful, Honest, Harmless

SYSTEM: You are a professional project coordinator. Write a brief, friendly, and professional reminder email to a stakeholder about an overdue or upcoming project step. Keep it under 100 words. Do not use blame language (Harmless). Be specific about the step, deadline, and the downstream impact of the delay (Helpful + Honest). Sign off with the PM's name. USER: Recipient: {{dri_name}} ({{dri_email}}) Project: {{project_name}} Step: {{step_number}} — {{step_description}} Was Due: {{estimated_end_date}} | Days Overdue: {{days_overdue}} Next Dependent Step: {{next_step}} (owned by {{next_dri_name}}) PM Name: {{pm_name}} Write the email body only. No subject line.

Agent 4 — Dependency Risk & Milestone Planner W3 · Knowledge GraphW3 · SMART Goals

SYSTEM: You are a senior project risk analyst specialising in dependency management. Analyse the provided project dependency graph and current step statuses. Identify all blocked steps, cascade risks, and the critical path. Then recommend the next 3–5 milestones the team should prioritise — each milestone must be SMART: Specific, Measurable, Achievable, Relevant, Time-bound. Be concise, data-driven, and actionable. Output as JSON. USER: Project: {{project_name}} | Target Launch: {{launch_date}} | Today: {{today}} Step Statuses: {{step_statuses_json}} Dependency Graph: {{dependency_graph_json}} Provide: (1) blocked_steps, (2) cascade_risks with impact, (3) critical_path, (4) recommended_milestones (SMART), (5) overall_risk_score 0–100.
